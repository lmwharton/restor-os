functions\nCREATE OR REPLACE FUNCTION get_user_company_id() RETURNS UUID AS $$\n    SELECT company_id FROM company_members\n    WHERE user_id = auth.uid() AND is_active = true LIMIT 1;\n$$ LANGUAGE sql SECURITY DEFINER STABLE;\n\nCREATE OR REPLACE FUNCTION user_belongs_to_company(check_company_id UUID) RETURNS BOOLEAN AS $$\n    SELECT EXISTS (\n        SELECT 1 FROM company_members\n        WHERE user_id = auth.uid() AND company_id = check_company_id AND is_active = true\n    );\n$$ LANGUAGE sql SECURITY DEFINER STABLE;\n\nCREATE OR REPLACE FUNCTION user_has_role(check_company_id UUID, required_role user_role) RETURNS BOOLEAN AS $$\n    SELECT EXISTS (\n        SELECT 1 FROM company_members\n        WHERE user_id = auth.uid()\n          AND company_id = check_company_id\n          AND is_active = true\n          AND role = required_role\n    );\n$$ LANGUAGE sql SECURITY DEFINER STABLE;\n\n-- Pattern applied to every company-scoped table:\n-- SELECT: any active member of the company can read\n-- INSERT: any active member can create (with company_id matching their membership)\n-- UPDATE: any active member can update rows in their company\n-- DELETE: owner only\n\n-- Example: job_schedules RLS\nALTER TABLE job_schedules ENABLE ROW LEVEL SECURITY;\n\nCREATE POLICY \"Members can view schedules\" ON job_schedules\n    FOR SELECT USING (user_belongs_to_company(company_id));\n\nCREATE POLICY \"Members can create schedules\" ON job_schedules\n    FOR INSERT WITH CHECK (user_belongs_to_company(company_id));\n\nCREATE POLICY \"Members can update schedules\" ON job_schedules\n    FOR UPDATE USING (user_belongs_to_company(company_id));\n\nCREATE POLICY \"Owners can delete schedules\" ON job_schedules\n    FOR DELETE USING (user_has_role(company_id, 'owner'));\n\n-- adjuster_reports: public read via access_token (no auth required for GET /public/report/:token)\nALTER TABLE adjuster_reports ENABLE ROW LEVEL SECURITY;\n\nCREATE POLICY \"Members can view reports\" ON adjuster_reports\n    FOR SELECT USING (user_belongs_to_company(company_id));\n\nCREATE POLICY \"Members can create reports\" ON adjuster_reports\n    FOR INSERT WITH CHECK (user_belongs_to_company(company_id));\n\n-- Public access via access_token is handled at the API layer, not RLS\n-- The GET /v1/public/report/:token endpoint uses a service-role key to bypass RLS\n```\n\n---\n\n## Part 5: API Endpoints (Grouped by Domain)\n\n### Auth & Company\n| Method | Path | Auth | Purpose | Priority Integration Architecture\n\n### Photo Scoping Pipeline (V1 Core -- Killer Feature)\n```\nClient uploads photo via presigned URL\n  -> POST /ai/photo-scope { photo_id, job_id }\n  -> FastAPI fetches photo from Supabase Storage\n  -> Sends to Claude Vision API with scoping prompt\n  -> Prompt includes S500/OSHA justification instructions\n  -> Parses structured Xactimate items + justifications\n  -> Stores in ai_scope_results\n  -> Streams results to client via SSE\n  -> User reviews: each line item shows justification references\n  -> User approves -> scope_entries created (with justifications JSONB populated)\n```\n\n**Justification Auto-Suggestion:**\nWhen Claude analyzes a photo and generates scope line items, the prompt instructs it to also return applicable industry Multi-Tenancy & Security\n\n### Tenant Isolation\n- Every data table includes `company_id` foreign key\n- PostgreSQL RLS policies enforce tenant boundaries at the database level\n- FastAPI middleware extracts `company_id` from the authenticated user's membership\n- No cross-tenant data access is possible even if API logic has bugs (defense in depth via RLS)\n\n### Authentication\n- Supabase Auth handles user registration, login, password reset, OAuth (Google)\n- JWTs issued by Supabase, validated by FastAPI on every request\n- Refresh tokens managed by Supabase client SDK\n\n### Authorization\n- Three roles: `owner`, `admin`, `tech`\n- Owner: full CRUD + billing + team management + delete operations\n- Admin: full CRUD + team management (no billing, no delete)\n- Tech: read + create + update on assigned jobs (no delete, no team management)\n- Role checked at the API layer via FastAPI dependencies\n\n### API Security\n- CORS: allowlisted origins only (Vercel deployment URLs)\n- Rate limiting: per-user, per-endpoint (stricter on AI endpoints)\n- Input validation: Pydantic models on every endpoint\n- SQL injection: prevented by parameterized queries (Supabase client)\n- File upload: type validation, size limits (10MB photos, 25MB audio)\n\n### Public Report Access\n- Adjuster/customer reports are accessed via unique `access_token` in the URL\n- Tokens are cryptographically random (32 bytes, hex-encoded)\n- Reports expire after 7 days by default\n- No authentication required for public report viewing\n- Public view shows limited data (no internal notes, no pricing, no company financials)\n- `viewed_at` timestamp recorded on first access\n\n---\n\n## Part 9: Frontend Architecture\n\n### App Structure & Routing\n\n```\n/                           -> Select photos -> [Analyze with AI] button\n2. Pulsing \"Analyzing...\" with streaming status text\n3. Line item cards fade in one-by-one (staggered animation)\n4. Each card shows:\n   - Xactimate code + description\n   - Quantity + unit\n   - Confidence bar: green (>80%), yellow (60-79%), red (<60%)\n   - Justification badges: \"IICRC S500 s10.3.2\", \"OSHA 29 CFR...\"\n   - [Approve] [Edit] [Reject] actions\n5. [Approve All Remaining] + [Add to Scope] bulk actions\n```\n\n**Card States:**\n- **Pending** (yellow border) -> review needed\n- **Approved** (green border, checkmark)\n- **Rejected** (red border, strikethrough, undo for 5s)\n- **Edited** (blue border, \"modified\" badge)\n\n**Justification Display:**\n- Each line item card has expandable \"Standards\" section\n- Shows linked IICRC S500, OSHA, or other standard references\n- Contractor can add/edit/remove justifications before approving\n- Justifications are included in PDF reports and adjuster reports\n\n**Error Handling:**\n- AI down -> \"Analysis unavailable. Photos saved -- analyze later.\"\n- Partial stream failure -> show received items + \"Retry remaining\"\n- Bad photo quality -> warning, proceed anyway or retake\n\n### Camera / Photo UX\n\n```\nUser taps \"Add Photo\"\n  -> <input type=\"file\" capture=\"environment\"> (native camera)\n  -> Client-side compression (4MB -> ~400KB via browser-image-compression)\n  -> Thumbnail generated locally for instant display\n  -> Upload via presigned URL\n  -> Upload progress with retry on failure\n```\n\n### Voice Architecture\n\n**Primary:** Web Speech API (free, offline-capable on many devices)\n**Fallback:** Deepgram (noisy environments, long dictation)\n\n**Voice Commands:**\n```\n\"next\"     -> advance wizard step\n\"back\"     -> previous step\n\"done\"     -> complete section\n\"approve\"  -> approve AI suggestion\n\"reject\"   -> reject AI suggestion\n```\n\n**Design Principles:**\n- Voice is enhancement, never the only path\n- Every voice field has keyboard input alongside\n- Noise detection: consecutive low-confidence -> suggest keyboard\n- Critical fields (phone, address): require explicit confirm\n\n---\n\n## Part 11: PWA & Mobile-First Design\n\n###
